I came back to the phone flag fix and tried to turn it into leverage instead of
just moving on. The narrow question was: are there other places where source
code looks like it is checking a value but is really reading a flag from one
instruction earlier?

The first brute-force search was too broad. Lots of assembly intentionally
loads an argument and branches on a previous compare. That is normal. The
interesting shape was narrower: a commented-out flag-setting instruction sitting
directly above a conditional branch. That was exactly the smell in the phone
delete guard, and it is rare enough that a triage helper can look for it without
turning into a siren.

Adding that check immediately pointed at `AI_Setup` in the legacy battle AI.
This was a nice little moment because the scanner was not wrong, but it also
was not a bug in the dramatic sense. The code had:

`cp EFFECT_EVASION_UP + 1`

then a commented-out compare against the next lower bound, then:

`jr z, .checkmove`

That works only because the gap between the stat-up and stat-down effect ranges
is currently a single effect. The previous compare leaves Z set for that single
gap value, so the branch skips it. Same trick for the +2 stat range. The current
behavior was intentional or at least inherited, and it was probably saving a
couple bytes in 1999-brained assembly style.

But the project has already added new effects. Focus Punch replaced one of the
old unused slots. Dragon Dance has special setup handling. This code should not
be clever about gaps between effect ranges. If another effect ever lands between
`EFFECT_EVASION_UP` and `EFFECT_ATTACK_DOWN`, stale-Z logic would start
treating the gap as stat-down setup. That is exactly the kind of bug that looks
like AI taste instead of code failure.

So I made it plain:

`cp EFFECT_ATTACK_DOWN`

`jr c, .checkmove`

and the same for `EFFECT_ATTACK_DOWN_2`. It costs a few bytes and buys back the
truth of the structure. The code now says "skip anything below the stat-down
range" instead of "remember which one magic gap value the previous compare made
zero."

This is not a sweeping change. It is the kind of maintenance I like here:
one brittle assumption turned into an explicit bound, and a triage helper taught
to notice the smell next time.
