# Audio Command Stack Walk

PowerShell says the local timestamp is `2026-04-27_2017`, so that is the filename timestamp. I opened the memory index first, not the code, because I wanted to avoid pretending a familiar hallway was new. The notes point hard at Boss AI, balance docs, Dragon work, map topology, and low-risk gameplay checks. Audio was sitting in the top-level directory with almost no smell from memory, so I walked toward it.

I listed the repo root next. The tree is loud with build products at the top level, but `audio/` is visually different: one clean noun, no obvious project-management wrapper around it. The worktree was already dirty in docs/outbox/decisions plus `Clean up.md`; I am not touching any of that. This is a journal-only pass unless the code makes a very small documentation breadcrumb feel unavoidable.

I opened the audio directory with a shallow recursive listing and immediately got the shape: a few machine-room files directly under `audio/`, then a long shelf of `audio/music/*.asm` song scripts. `audio.asm` is the doorframe. It includes `audio/engine.asm`, then trainer encounter music, pointer tables, one empty track, cry pointers, SFX pointers, and then bank-grouped music includes. The bank assertions stood out more than the songs did. This file is not a composer workspace; it is a linker seating chart.

I also ran an overbroad `rg` for music terms and got punished with thousands of lines. That made me click away from the song files for now. The interesting thing is not the notes yet; it is the tiny language those files speak: `channel_count`, `channel`, `sound_call`, `sound_loop`, `sound_ret`, `toggle_noise`, `tempo`. I want the interpreter before the sheet music.

I opened `audio/engine.asm` at the top and it starts with exactly the warning I wanted: "The entire sound engine. Uses section `audio` in WRAM. Interfaces are in bank 0." The first routines are `_InitSound`, `MusicOn`, `MusicOff`, and `_UpdateSound`. The update loop is once per frame, walks eight channel structs, decrements note duration, calls `ParseMusic` when a note expires, applies pitch slide, vibrato, noise handling, mutes music under priority SFX, then writes `wVolume` and `wSoundOutput` to hardware. This is not decorative audio code. It is a tiny scheduler with four music channels and four SFX channels sharing the same body plan.

`audio/music_pointers.asm` felt like a census ledger, not logic. It says entries correspond to `MUSIC_*` constants and emits `dba` triples, so every song identity is bank plus address. `audio/notes.asm` is smaller and stranger: a frequency table with one silent entry and two octaves worth of words. I clicked away because it explains pitch material but not control flow.

I found the macro file through `includes.asm`, which includes `macros/scripts/audio.asm` early, before the constants and before legacy aliases. That is the composer-facing grammar. I also noticed `macros/legacy.asm` still maps old names like `loopchannel`, `callchannel`, and `endchannel` onto `sound_loop`, `sound_call`, and `sound_ret`. That explains why the modern files read consistent without making the old project vocabulary disappear.

I made a small shell mistake trying to include `ram/*.asm` in one `rg`; PowerShell passed it badly and `rg` complained about `ram\*.asm`. I left that in my head as a reminder that this repo is Windows-hostile in exactly the places old disassemblies tend to be. The useful hit was still there: `constants/audio_constants.asm` defines the channel struct offsets while `macros/ram.asm` defines the WRAM shape with comments. The channel struct is the anchor. It carries music id, bank, flags, address, last address, condition, duty, envelope, frequency, pitch, octave, transposition, duration, loop count, tempo, tracks, vibrato, slide target, pitch offset, and several unnamed fields. The unnamed fields are a scent, but not today's trail.

`macros/scripts/audio.asm` is pleasingly direct. Notes below `$d0` are packed pitch/length nybbles. `$d0` through `$d7` are octave commands. `$d8` is `note_type`, `$da` is `tempo`, and the high tail ends with flow control: `$fa` condition, `$fb` conditional jump, `$fc` jump, `$fd` loop, `$fe` call, `$ff` return. The file is doing the good kind of magic: it lets music read like notation while still making the bytecode visible.

Then I opened `audio/engine.asm` around `ParseMusic`. It parses until it sees either a playable note or the end of a stream. If the byte is below `FIRST_MUSIC_CMD`, it is note data. If it is `$d0` or above, it dispatches through `MusicCommands` and loops back to parse more. That means commands like tempo or octave are state changes, not frame-consuming events. A note is what finally hands control back to the frame update.

The `sound_ret` behavior is cleaner than I expected. `$ff` inside a subroutine routes to `Music_Ret`, clears `SOUND_SUBROUTINE`, and restores `CHANNEL_LAST_MUSIC_ADDRESS`. `$ff` outside a subroutine shuts the channel down, marks a rest, and clears music id and bank. So the same byte is both "return" and "stream ended," depending on one flag in the channel struct. That is compact and a little dangerous, but not ugly.

I followed the public doorway into `home/audio.asm`. This file exists because the engine lives in a switchable bank but callers need stable bank-0 wrappers: `InitSound`, `UpdateSound`, `PlayMusic`, `PlayCry`, `PlaySFX`, `WaitSFX`, map music helpers. The wrappers save the current ROM bank, switch to the audio engine bank, call the real routine, then restore the bank. `_LoadMusicByte` is the tiny courier: set bank `a`, read `[de]` into `wCurMusicByte`, then switch back to `BANK(LoadMusicByte)`. I clicked away from broad caller search here because every menu, animation, field move, and movie hits audio; the interface file was more informative than the call graph.

`_PlayMusic` is the conversion point from id to channel structs. It stores `wMusicID`, indexes the `Music` pointer table by three-byte `dba`, loads the song header byte, extracts the channel count from the top bits, then repeatedly calls `LoadChannel` and `StartChannel`. After that it clears the odd little jump-condition and noise-sample globals, then calls `MusicOn`. That makes the `channel_count`/`channel` header macros make sense: one packed header byte, then channel id/address pairs.

`_PlayCry` and `_PlaySFX` reuse the same channel loader instead of having separate engines. Cries load from `Cries`, set `SOUND_CRY`, force pitch offset, use `wCryPitch` and `wCryLength`, may pan toward the monster's side in stereo, and temporarily max volume. SFX load from `SFX`, clear the four SFX hardware channels first, set `SOUND_SFX`, and start channels five through eight. `home/audio.asm` also adds a priority gate before `_PlaySFX`: if a sound is already playing, a lower-priority new id is ignored. The comment says SFX are ordered highest to lowest, and the compare matches that.

I opened `ram/wram.asm` after that because I wanted to see whether audio state was isolated. It is. `SECTION "Audio RAM", WRAM0` starts with `wMusicPlaying`, then eight channel structs, then the transient current-track fields, global music id/bank/noise/fade/cry/SFX state, then `wAudioEnd`. `wMapMusic` and `wDontPlayMapMusicOnReload` sit just after `wAudioEnd`, which feels like an intentional boundary: engine internals end, map-level policy begins.

The two song examples I opened were not random, but they were not the whole music catalog either. `audio/music/successfulcapture.asm` is only three channels of setup and then `sound_jump`s into `Music_WildPokemonVictory_*` bodies. `audio/music/johtowildbattlenight.asm` does the same kind of reuse with different tempo, offsets, vibrato, and a small lead-in on channel 2 before looping into the normal Johto wild battle bodies. That is a better sign than a fully copied night battle song would have been. The music language is being used for composition, not just transcription.

I went into noise last because `toggle_noise` had been sitting there as an unresolved word. `audio/drumkits.asm` is a table of drum-kit tables, each kit mapping note pitches to little `noise_note` streams like `Snare1`, `Triangle2`, `HiHat1`, and `Kick1`. `GetNoiseSample` only accepts the note if the current channel modulo four is channel 4; then it picks `wMusicNoiseSampleSet` or `wSFXNoiseSampleSet`, uses the note's pitch nybble as an index into the selected kit, and stores that drum sample pointer in `wNoiseSampleAddress`. `ReadNoiseSample` later walks three-byte noise-note structs, delays by duration, and feeds envelope/frequency into the current track fields until it reaches `sound_ret`.

That made `toggle_noise` feel less like a flag and more like an instrument change. In normal music, it selects which kit channel 4 should interpret its pitches through. In SFX, `sfx_toggle_noise` does the same for the SFX noise lane. The macro comment in `Music_ToggleNoise` says it "can't be used as a straight toggle since the param is not read from on->off"; that is the kind of comment future me would be glad to have. The command name is a little misleading, but changing it would fight the disassembly's inherited vocabulary and probably make the old music files harder to compare upstream.

I opened `audio/wave_samples.asm` on the way out because it was included next to `drumkits.asm`. It is not control flow at all, just 32-nybble wave patterns with a comment telling you to plot them as line charts. Nice file. It gives the wave channel its material the way `drumkits.asm` gives noise its material. I clicked away because the command stack was the thread, not audio synthesis as a whole.

Stopping here. The part I wandered into is smaller than it first looked: `audio.asm` seats banks and includes; `macros/scripts/audio.asm` defines the byte language; `home/audio.asm` is the bank-safe interface; `audio/engine.asm` loads ids into channel structs and interprets script bytes once per frame; `ram/wram.asm` owns the state; `audio/drumkits.asm`, `audio/notes.asm`, and `audio/wave_samples.asm` supply material. The nicest thing I found is that SFX, cries, and music are not three engines. They are one channel machine with different flags, loaders, and policies.
