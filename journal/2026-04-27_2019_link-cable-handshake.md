# Link Cable Handshake Walk

PowerShell says the local timestamp is `2026-04-27_2019`, so that is the timestamp in this filename. I checked memory and the current worktree before opening code. The recent paths are crowded with Boss AI, Dragon edits, balance docs, map topology, and the audio command stack from the last walk. The worktree is already dirty in docs, outbox, decisions, `Clean up.md`, and the prior audio journal. I am leaving all of that alone and adding only this entry.

I went looking for link and serial code because it appeared in the tree but not as a recent memory lane. `rg --files` gave me `engine/link/link.asm`, several mystery gift and time capsule files, `home/serial.asm`, `home/printer.asm`, and `constants/serial_constants.asm`. The shape already felt different from normal gameplay code: fewer small feature files, more protocol machinery.

I listed `engine/link/` and it is mostly one giant file: `link.asm` is about 43 KB, with little satellites for init lists, time capsule, mystery gift, and waiting text. That made me open the giant file first, even though I expected it to be unpleasant. Sometimes the biggest file is the source of truth because the smaller ones only make sense after you see the state machine they orbit.

`LinkCommunications` begins like a stage reset, not a protocol: delay, clear tilemap, clear sprites, load fonts, load the trade border, draw a "Please wait" box, and set `wLinkByteTimeout` to `SERIAL_LINK_BYTE_TIMEOUT`. Only then does it branch on `wLinkMode`. `LINK_TIMECAPSULE` goes to `Gen2ToGen1LinkComms`; everything else goes to `Gen2ToGen2LinkComms`. The first surprise is that the time capsule is not just another room mode. It has its own transfer width, party conversion path, and patch-list handling.

I opened `constants/serial_constants.asm` because the raw bytes in `link.asm` were already too symbolic to read cold. The constants are a small map: link modes 0 through 3, serial high nybbles for time capsule/trade/battle, clock status values, patch-list sentinels, preamble lengths, random-number lengths, mail preambles, and the `$5000` byte timeout. The three special byte values are the important ones: `$fd` starts data, `$fe` means no data, `$ff` ends a patch-list part and is also the replacement byte for `$fe` inside patchable data.

Then I opened `home/serial.asm`, and it made the link code feel less mystical. The serial interrupt either delegates to the printer path or handles link bytes. If the connection is not established, it watches `rSB` for `USING_EXTERNAL_CLOCK` or `USING_INTERNAL_CLOCK`, records the status in HRAM, and primes the next send byte as `$fe`. Once established, it copies `rSB` to `hSerialReceive`, writes `hSerialSend` back to `rSB`, and kicks the serial control register again for external-clock peers. This is a byte pump with a little handshake bolted on the front.

I opened HRAM and WRAM next because the names were doing too much work. HRAM has exactly the serial scratch I expected: `hSerialReceivedNewData`, connection status, ignore-initial-data flag, send byte, receive byte. WRAM has a small overlapped link-action area where `wOtherPlayerLinkMode`, `wOtherPlayerLinkAction`, and `wBattleAction` share space, then player action, sync buffers, frame timeout, and byte timeout. The overlap made me pause. It is economical, but it also means these names are only honest inside the right link phase.

I stayed in `home/serial.asm` for `WaitLinkTransfer` because the top-level link file kept calling it like a magic word. It initializes the other player's action to `$ff`, loops `LinkTransfer` until a real action appears, then does a receive window and an acknowledge window before copying the other player's action into `wOtherPlayerLinkMode`. That name is a little slippery: during the wait, the low nybble is an action, while the high nybble identifies time capsule, trade center, or battle. The code makes that explicit in `LinkTransfer`, which picks a high nybble based on `wLinkMode`, adds it to `wPlayerLinkAction`, and sends the combined byte.

`Serial_ExchangeSyncBytes` is smaller and more disciplined. It exchanges two bytes from `wLinkPlayerSyncBuffer` into `wLinkReceivedSyncBuffer`, ignoring the first transfer. The comment says it checks that both players entered the same Cable Club room. I clicked away after that because the byte-level loop was clear enough: the real weirdness is not sending bytes, it is making arbitrary party/mail data safe to send when `$fd`, `$fe`, and `$ff` have protocol meanings.

So I opened `FixDataForLinkTransfer`. It first writes a preamble into `wLinkBattleRNPreamble`, then fills ten random bytes while rejecting values at or above `$fd`, then initializes `wPlayerPatchLists`. After that it walks two patchable areas inside `wLinkData`, finds any `$fe` bytes, records their one-based offsets in the patch list, and replaces those bytes with `$ff`. This is the sort of low-level compromise that makes old link protocols feel hand-built: the data stream reserves certain byte values, so real data has to travel with a repair kit.

`Link_PrepPartyData_Gen1` is where the time capsule earns its separate path. It writes the preamble, player name, party count/species list, converts species through `ConvertMon_2to1`, copies six party structs in Red format, copies OT names and nicknames, and has special Magnemite/Magneton handling to erase Steel typing back to Electric. I clicked away before fully auditing every copied stat field because the main observation was already strong: Gen 1 compatibility is not a filter at the edge, it reshapes the whole party payload.

`Link_PrepPartyData_Gen2` is quieter. It writes the same preamble, then copies player name, party count/species/end, player id, party structs, OT names, and nicknames directly. The trade center adds mail: a five-byte `$20` preamble, messages from SRAM, metadata from SRAM, then separate patching for `$fe` in mail text and metadata. Mail text uses `$21` as the replacement byte, while metadata uses the normal `$ff` patch replacement plus offsets. I clicked away because the data marshaling pattern had repeated enough to understand the design.

I opened `engine/link/init_list.asm` because the name sounded like it might be a tidy satellite. It begins with `; This entire function is useless.` That is blunt enough to end the visit quickly. The body sets old list/name/item pointers based on `wInitListType`, but the opening comment is the useful part today: some of the satellites are historical debris, not active structure.

I jumped near the bottom of `engine/link/link.asm` because `data/events/special_pointers.asm` showed the map scripts call link routines as specials. That section is the receptionist machinery. `SetBitsForLinkTradeRequest` and `SetBitsForBattleRequest` store the chosen room in both `wPlayerLinkAction` and `wChosenCableClubRoom`. `SetBitsForTimeCapsuleRequest` is different: it pokes serial registers directly with `USING_INTERNAL_CLOCK` and leaves the chosen room as null. Again, time capsule is not merely "trade center with old rules"; it has an older handshake shape.

`WaitForLinkedFriend` is the lobby loop. If there is a link action, it tries to establish the connection with internal/external clock bytes, then repeatedly alternates internal and external starts until HRAM says a clock was found or the timeout expires. The Virtual Console hooks are not decoration here. One hook explicitly fakes `hSerialConnectionStatus` to let the player get past the receptionist's "Please wait," and it asserts the HRAM address and `USING_INTERNAL_CLOCK` value. That is a sharp preservation note: VC wireless compatibility depends on exact memory placement.

The map script in `maps/Pokecenter2F.asm` made the user-facing order legible. Trade and battle receptionist scripts set the requested room, ask, wait for the friend, require saving, quick-save, check timeout, reject first-gen peers for those rooms, compare selected rooms, then walk the player into the room. Time Capsule first checks local compatibility, then accepts first-gen peers as the good path. This is where the protocol turns back into game design: the receptionist text is just a friendly wrapper around a fragile two-machine agreement.

I opened `CheckTimeCapsuleCompatibility` because the script branches on its exact return values. It rejects Johto Pokemon, mail, and moves after `STRUGGLE`, saving the incompatible Pokemon or move name for text. The Gen 1 limit is not hidden in conversion; the player is stopped before entering. I clicked away because the check is straightforward and the more interesting thing is that compatibility is enforced twice: once before entry, and again later by `ValidateOTTrademon` during trade.

`CheckBothSelectedSameRoom` uses `Link_EnsureSync`, compares the received low nybble with `wChosenCableClubRoom`, and sets `wLinkMode` to chosen room plus one. That plus-one is why cable club room constants start at trade center 1 while link mode constants start with time capsule 1 and trade center 2. The naming is not beautiful, but the arithmetic is intentional. I would not rename this casually without writing a decision file first.

I opened `home/vblank.asm` only because `WaitForLinkedFriend` and the timeout checks keep setting `hVBlank` to `VBLANK_SOUND_ONLY`. `VBlank_Serial` normally updates graphics, joypad, updates the serial poll, and updates sound. In the link wait phases they do not want the full normal frame behavior. The removed printer poll used to handshake when the printer connection was open, once every six frames, by sending `$88` with the internal clock. I clicked away from printer because this walk is link cable, but now I understand why the serial interrupt had that printer branch at the top.

I ended at `Link_EnsureSync`. It adds `$d0` to a small value, writes the same byte twice to `wLinkPlayerSyncBuffer`, switches vblank to sound-only, exchanges sync bytes until it sees a received byte with high nybble `$d0`, then returns the low nybble. That is the small version of the whole system: reserve a high nybble as a marker, pack the real meaning into the low nybble, loop until the other machine speaks the same tiny language.

Stopping here. I did not get into the trade menu UI or battle record saving because the handshake thread had already carried me from map script, to specials, to HRAM/WRAM state, to serial interrupt, to party payload patching. The part that matters is this: link is not one feature. It is a stack of small agreements about which bytes are control bytes, which nybbles mean room/action, which clock owns the next transfer, and which old-game facts have to be preserved before the player is allowed through the door.
