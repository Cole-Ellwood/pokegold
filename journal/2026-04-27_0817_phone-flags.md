I started with the recent bug-fix lane because the last commits were all in the
same family: stale text, source drift, things that look harmless because they
are "only words" until they leak into the player experience. The release smoke
was green, the docs navigation check was green, and the triage tool had no fresh
leads. That usually means either the repo is quiet or I am standing in a place
where the bugs are not noisy.

So I went into the phone system.

The phone code has the nice old-engine feeling where everything is a compact
table relation: contact constants, trainer classes, map ids, time gates, caller
scripts, rematch events, and landmark text all braided together. I expected a
mismatch there. Something like "Jose says Route 27 but the contact table says
Route 26," or a caller setting an event that no map clears. The rematch graph
actually looked cleaner than I expected. Every ready-for-rematch event I checked
had the expected map-side check and clear, and the swarm calls seemed to point
at real swarm encounter tables.

The weird little thing was not in those big relationships. It was a commented
out `and a` in `CheckCanDeletePhoneNumber`.

That line bothered me because it was exactly the kind of assembly that can sit
there for years wearing the shape of logic without actually doing it. `ld a, c`
does not refresh flags. The next instruction was `ret nz`. So the code was not
testing the trainer class it had just loaded; it was trusting whatever flags
survived the helper call.

The funny part is that this did not look like a player-visible bug today.
`GetCallerTrainerClass` calls `AddNTimes`, and `AddNTimes` returns with Z set
for the contact indexes this path uses. The rest of the routine then still
returns a nonzero `c` for trainer contacts, so the Pokegear menu still offers
DELETE where it should. The behavior was right for the wrong reason, or at least
right through a path no one should have to keep in their head.

I restored the explicit `and a` and put a smoke check around it. That feels like
the correct size of fix: no architecture, no new abstraction, just removing a
stale flag dependency before a future edit changes the helper's flag behavior
and turns "harmless commented instruction" into a real menu bug.

I also noticed the Bike Shop contact still points at `OAKS_LAB` in the contact
table. That looked wrong for about thirty seconds, then less wrong after tracing
the special-call path and comparing upstream. It is vanilla filler data for a
contact you cannot manually call through the normal callee path. I did not touch
it. That is worth remembering because it is the kind of thing a future session
could easily "fix" and accidentally make noise without solving anything.
