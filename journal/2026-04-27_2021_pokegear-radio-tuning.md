# Pokegear Radio Tuning Walk

PowerShell says the local timestamp is `2026-04-27_2021`, so that is the timestamp in this filename. I checked memory first because the project has a lot of recent gravity: Boss AI, Dragon balance, map topology, docs navigation, the audio command stack, and the link cable handshake. The worktree is still dirty with existing docs/outbox/decisions changes, `Clean up.md`, and the prior two new journal files. I am leaving all of that alone and adding only this entry.

I went looking for phone, radio, and Pokegear files because they appeared in broad search but did not feel like a recent lane. The file list immediately split into `engine/phone/`, `engine/pokegear/`, `data/phone/`, `data/radio/`, and constants for both systems. I clicked away from phone for now because it is sprawling through trainer scripts and overworld calls. Radio looked smaller and more contained: a card in the Pokegear, a station script engine, channel music, and a few data tables.

I opened `engine/pokegear/pokegear.asm` first because radio is a card inside that shell. The file starts with card ids for clock, map, phone, and radio, then jumptable states for clock/map/phone/radio init and joypad. `PokeGear` itself is a menu loop: save options and menu state, turn off text scroll, initialize graphics, then each frame update time, read joypad, dispatch `PokegearJumptable`, play sprite animations, and delay. The exit path calls `ExitPokegearRadio_HandleMusic`, which told me radio has side effects that survive the card unless explicitly unwound.

I opened `constants/radio_constants.asm` next because the radio ids are shared by three systems: channel songs, the radio-show jumptable, and the tuning table in the Pokegear file. The first ten constants are station ids the user can tune to: Oak's talk, Pokedex show, Pokemon music, Lucky Channel, Places and People, Let's All Sing, Rocket Radio, Poke Flute, Unown, Evolution. After that the constants become internal segment ids. That split matters. A station is not one text block; it is a little script made of segments.

The WRAM slices made the shape more obvious. The Pokegear overlay stores phone cursor state, map cursor state, then `wPokegearRadioChannelBank`, `wPokegearRadioChannelAddr`, and `wPokegearRadioMusicPlaying`. A later overlay stores the radio program state: `wCurRadioLine`, `wNextRadioLine`, `wRadioTextDelay`, `wNumRadioLinesPrinted`, `wOaksPKMNTalkSegmentCounter`, and `wRadioText`. The two sets are separate for a reason: one remembers which station function to call; the other remembers where the current broadcast script is inside its lines.

I ran a focused search for radio symbols and got the spine: `PokegearRadio_Init`, `PokegearRadio_Joypad`, `_UpdateRadioStation`, `UpdateRadioStation`, `RadioChannels`, station loaders, music restart helpers, `NoRadioStation`, `PlayRadio`, and then `engine/pokegear/radio.asm` with `PlayRadioShow`, `RadioJumptable`, `PrintRadioLine`, `RadioScroll`, `StartRadioStation`, and `NextRadioLine`. I clicked away from the giant search output quickly because the names were enough. The next useful thing was the tuning path, not every station.

I opened the Pokegear jumptable region. Radio is one of the last two states: `PokegearRadio_Init` and `PokegearRadio_Joypad`. Clock/map/phone states call `ExitPokegearRadio_HandleMusic` when they become active, which explains why the Pokegear shell has to know about radio even when the radio card is not open. Switching away from a station is not only a UI change; it may have to restart map music.

`PokegearRadio_Init` draws the tilemap, creates a tuning knob sprite, calls `_UpdateRadioStation`, then advances to joypad state. `PokegearRadio_Joypad` is stark: B exits, Left switches to phone/map/clock depending on owned cards, and otherwise if a saved radio farcall pointer exists it calls it every frame. There is no A button to "play" a station inside the Pokegear. Tuning selects a station, and the station routine advances itself while the card sits open.

The knob path is neat. `AnimateTuningKnob` lives near the radio tables because the sprite animation function calls into this file. Up and down adjust `wRadioTuningKnob` by two, bounded at 0 and 80, then fall through into `UpdateRadioStation`. The comment on `RadioChannels` decodes the weird values: frequency value equals four times the in-game frequency minus two. So `16` means 4.5, `28` means 7.5, `78` means 20.0. The knob stores the scaled value, not the display frequency.

`UpdateRadioStation` walks `RadioChannels` looking for an exact knob match. If it misses, it calls `NoRadioStation`. If it hits, it jumps to a station selector and expects `de` to come back pointing at the station name. The selector can still refuse reception based on region, time of day, expansion card, Rocket signal, or landmark. This is why the same knob value can be a station, static, or a different station depending on where the player is.

I liked the geography rules more than I expected. Johto gets Oak/Pokedex, Pokemon Music, and Lucky Channel. Kanto gets Places and People, Let's All Sing, and with the expansion card, Poke Flute. Ruins of Alph gets Unown only if the player landmark is exactly Ruins of Alph. The Evolution signal is a Lake of Rage/Mahogany/Route 43 special while the Rocket signal flag is set. It reads like radio propagation but is just landmark tests and flags.

The station loaders are repetitive on purpose. Each one sets `wCurRadioLine` to the station id, clears `wNumRadioLinesPrinted`, stores the farcall target as `BANK(PlayRadioShow):PlayRadioShow`, and returns a station-name pointer. That storage step is the hinge between tuning and broadcast: the Pokegear joypad state later farcalls whatever got saved in `wPokegearRadioChannelBank/Addr`.

I opened `data/radio/channel_music.asm` because `StartRadioStation` indexes it by radio id. It is a tiny table: Pokemon Talk, Pokemon Center, title, Game Corner, Viridian City, bicycle, Rocket overture, Poke Flute channel, Ruins of Alph radio, Lake of Rage Rocket radio. I clicked away fast because the table is clear and the audio engine was already a previous walk. For this lane, the important point is that station ids drive both text scripts and music.

I opened `engine/pokegear/radio.asm` at the top. `PlayRadioShow` first checks whether Team Rocket has occupied the radio tower; if they have, and the player is in Johto, and the current station is one of the normal low ids, it overwrites `wCurRadioLine` with `ROCKET_RADIO`. Then it jumps through `RadioJumptable`. This is an elegant hijack. The tuning table does not need to know the tower plot state for every normal station. The broadcast engine can seize the program after tuning has already selected something plausible.

`RadioJumptable` is larger than the station list because it includes every segment. The constants file had already prepared me for that, but seeing the table made it real: Oak's talk runs from `$00` into `$0a` through `$11`, Lucky Channel runs through a long band, Rocket Radio has ten pieces, and `RADIO_SCROLL` is just another jumptable entry. The radio show is a state machine where the state variable is literally "which line routine runs next."

`PrintRadioLine` and `RadioScroll` are the broadcast rhythm. `PrintRadioLine` remembers the next segment in `wNextRadioLine`, copies or prints `wRadioText`, sets the current line to `RADIO_SCROLL`, and gives the text a delay of 100 frames. `RadioScroll` counts that delay down, restores `wCurRadioLine` to the next segment, copies the bottom line up once enough lines have printed, and clears the bottom line. The radio is not streaming text; it is alternating between "print a segment" and "wait/scroll."

I followed Oak's Pokemon Talk because it is the first and most alive-looking show. It starts by setting `wOaksPKMNTalkSegmentCounter`, starts station music, prints intro lines, then chooses a random route from `data/radio/oaks_pkmn_talk_routes.asm`. It searches `JohtoGrassWildMons`, skips percentages, picks a time of day other than darkness, then chooses one of the middle three Pokemon from that route's encounter table. That is a wonderful little editorial rule: Oak's show samples real wild data, but it avoids the edges of the encounter list.

After picking the Pokemon, Oak's show copies the Pokemon name, converts the map group/number into a landmark name, then alternates Oak and Mary text. Later segments pick random adverbs and adjectives from 16-entry tables. The constants file had a note that these table sizes are powers of two so no rejection sampling loop is needed. That comment earned its keep here; the code just masks random bits and indexes directly.

I clicked away from the rest of Oak's tables when it started becoming content rather than machinery. The next interesting detour was `OaksPKMNTalk10` through `OaksPKMNTalk14`, where it restarts the Pokemon Channel music, prints a little channel-identification sequence, then returns to Pokemon Talk music and loops back toward the random-route segment. This station is not one linear script. It has a bumper.

`CopyRadioTextToRAM` sent me briefly to `home/battle.asm`, which is not where I expected radio support to live. `FarCopyRadioText` reads a far text pointer, banks over, and copies exactly two screen-widths into `wRadioText`. `home/text.asm` has `RadioTerminator`, a tiny `text_end` provider. I clicked away from both because they are generic helpers, but the placement is odd. Radio text copying sits near battle helpers, probably because bank-0 utility space was convenient, not because the concept belongs there.

I opened `ExitPokegearRadio_HandleMusic` because it kept showing up whenever the UI leaves the radio card. It checks `wPokegearRadioMusicPlaying`: `RESTART_MAP_MUSIC` means call `RestartMapMusic`; `ENTER_MAP_MUSIC` means call `PlayMapMusicBike`; either way it clears the flag. This is the radio's cleanup contract. Stations are allowed to interrupt overworld music, but they must leave a breadcrumb saying how to restore the world.

Then I looked at `PlayRadio`, which is the map-object version of the same system. It saves text-scroll options, calls a station loader by index from `PlayRadioStationPointers`, draws a textbox with the station name in quotes, waits 100 frames, then loops the saved radio farcall until A or B is pressed. This path reuses the exact same station loaders and `PlayRadioShow`; a house radio and the Pokegear radio are different shells around the same broadcast core.

The map-script route confirmed it. `COLL_RADIO` tiles map to `Radio1Script`, which opens text, sets `MAPRADIO_POKEMON_CHANNEL`, calls special `MapRadio`, and closes. `Radio2Script` is the same thing for Lucky Channel. `MapRadio` itself only copies `wScriptVar` into `e` and farcalls `PlayRadio`. That is a clean script API: map scripts choose the station class, engine code owns the broadcast.

I followed `RadioTowerRocketsScript` because Rocket Radio kept appearing as a global override. That standard script sets `ENGINE_ROCKETS_IN_RADIO_TOWER`, changes several Goldenrod/Mahogany radio-tower events, triggers a special phone call, and moves Mahogany's scene forward. I clicked away from map plot details there. The important part for the radio engine is that the Rocket takeover is a single flag checked at broadcast time, not a special station table rewrite.

I stopped at the station-name strings: Oak's talk, Pokedex Show, Pokemon Music, Lucky Channel, an unknown `?????`, Places and People, Let's All Sing, and Poke Flute. That little list is the public face of the whole mechanism I just walked through. Behind each label is a tuner value, region gate, station id, saved farcall pointer, broadcast-state id, text-copy helper, scroll delay, and music restore flag.

Stopping here. I did not read the whole Pokegear file and I deliberately left phone alone. The radio part was enough: the codebase treats radio as a tiny broadcast VM. Tuning chooses a farcall; the farcall advances a segment-state machine; segment routines pull real game data when they need flavor; and music cleanup is tracked separately so the overworld can recover after the player closes the radio.
