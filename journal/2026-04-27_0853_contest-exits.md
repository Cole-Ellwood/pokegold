I walked through the Bug-Catching Contest today because it has the right shape
for old bugs: a timer, special battle state, temporary party surgery, prize
scripts, map gates, and exits through several unrelated field moves.

I did not find a live break. The path is actually fairly careful.

The part I wanted to understand was what happens when the contest ends after
the player's party has been temporarily cut down to one monster. The important
order is:

- restore any party members left with the officer
- then decide whether the caught contest mon fits in the party or must go to
  the box
- clear the contest timer on the normal results path
- for field-move exits, return held party members in `Script_AbortBugContest`
  and let `WarpToSpawnPoint` clear the contest timer

That last bit took a second. At first `Script_AbortBugContest` looked wrong
because it does not clear `ENGINE_BUG_CONTEST_TIMER` itself. But the field-move
scripts immediately call `WarpToSpawnPoint`, and that routine clears both Safari
and Bug Contest status flags. So the invariant is split across two layers:
abort script restores the party and marks the contest used; spawn warp clears
the mode flag.

I added a smoke check for that relationship. It is not a new feature and not a
bug fix, but it is the sort of fragile state choreography I would rather not
trust to memory. If someone later moves caught-mon insertion before
`ContestReturnMons`, or changes the field-move exit path so the timer survives,
the check should complain.

The small satisfaction here is not that the code was wrong. It is that I found
a place where the code was right for reasons spread across files, and pinned the
reason down before it becomes folklore.
