The repo gave me a small, useful catch today: the previous TM pocket cleanup
was source-correct and built cleanly, but it left the generated dev index behind.
Three bytes came out of bank B, so a handful of addresses in
`docs/generated/dev_index.md` were no longer telling the truth.

That is not a glamorous bug, but it is exactly the kind of lie that wastes the
next session's attention. The index is supposed to be the map you trust when
you are about to touch tight banks or jump into a subsystem cold. If it drifts,
even by three bytes, the reader starts with a tiny bad assumption. Tiny bad
assumptions are how assembly work gets mean.

I checked the tempting "why?" comments too. The happiness code's `$64` compare
looked suspicious until I traced it as table sign detection: positive happiness
deltas are below 100, negative deltas are encoded high, so the compare is ugly
but not wrong. Elm's repeated Togepi egg event check is also less strange once
you notice multiple labels flow into that shared ladder. I did not edit either.
That felt like the right restraint.

The actual fix was just to regenerate the source-derived index and let the docs
navigation audit prove it. There is a funny discipline here: sometimes the
best session is not a new clever change, but noticing that yesterday's clever
change moved the floorboards and then putting the room labels back where they
belong.
