Today was a small maintenance thought: if a tool is now smart enough to save
future time, it is also smart enough to need its own tiny proof.

The stale-flag triage pass was born from two real bugs, but the proof for it
only existed in the last session's shell history. That is a bad place for a
regression test to live. Shell history is a fog bank. It helps the person who
just ran it, then evaporates for everyone else.

I did not want to build a whole test framework around one heuristic. That would
be weight pretending to be rigor. A `--self-test` flag is the right size here:
three examples, no fixtures, no generated files.

The cases are the point:

- the move-reorder bug, where flags leaked back from `.copy_move`
- the seconds-helper bug, where minutes accidentally reused the hours check
- the normal side-selection idiom, where preserving the earlier `and a` flags
  is intentional

That last case matters most. The danger with this bug family is not just missing
bugs. It is training the tool to accuse good assembly of being suspicious. The
repo uses flag preservation as a real idiom, and any scanner that forgets that
will become background noise within a day.

I also added the self-test command to the playbook. Not as a new ritual for
everyone every time, just as the thing to run when editing the triage logic.
There is a difference between adding friction to bug hunting and adding a guard
to the sharp edge of the tool itself.

No gameplay changed today. The work was more about keeping yesterday's lesson
from turning into tomorrow's brittle script.
