# Boss AI Debugger Rule Map

Status: Phase-1 foundation for full ROM scoring traces.

The debugger rule map is generated by `tools/boss_ai_debugger/rule_map.py` and
stored at `audit/boss_ai_debugger/rule_map.json`.

## Commands

Build the stored artifact:

```powershell
python -m tools.boss_ai_debugger rule-map build --json-out audit\boss_ai_debugger\rule_map.json
```

Check that the stored artifact still matches source:

```powershell
python -m tools.boss_ai_debugger rule-map check
```

## Current Contract

Each rule entry records:

- `rule_id`: stable semantic id used by future trace events
- `source_file`: Boss AI source path
- `source_label`: asm label
- `line`: source line at generation time
- `parent_label`: enclosing `BossAI_` label for local labels
- `classification`: `platform_boundary`, `public_info`, `haki_exception`, or
  `internal`
- `public_reads`: public WRAM/source-table hints relevant to the rule

The check validates required Boss AI labels, duplicate ids, allowed
classifications, source hashes, public-read hints, and stored/current label
shape.

## Why This Exists

Line numbers are too fragile for a state-of-the-art debugger. The full scoring
trace phase needs event ids that survive source movement and still point back to
readable asm labels. This map is the bridge between trace bytes, source labels,
audit invariants, and future counterfactual reports.
