The "preserves other registers" claim is misleading: a is clobbered (caller's a after farcall = target's exit c due to lines 13-16 + 24-27), and the caller's hl is gone — killed by the macro's pre-call ld a, BANK :: ld hl, target before the target ever runs.

Suggested change

; FarCall_hl::
; Cross-bank call dispatcher invoked via `rst FarCall` (rst $08).
;
; ENTRY (set by `farcall`/`callfar` macros at the call site,
;        BEFORE this routine runs — so caller's a/hl are gone):
;   a  = BANK(target)
;   hl = target address
; EXIT (back at the caller after target returns):
;   a  = TARGET'S EXIT c       ; NOT target's exit a — see note below
;   hl = clobbered             ; set by the macro before `rst FarCall`
;   bc = target's exit bc      ; preserved through wFarCallBC stash
;   de = target's exit de
;   f  = target's exit f
;
; Why a is clobbered post-call: lines 13-16 stash target's b/c into
; wFarCallBC; the trailing `ld a, [wFarCallBC + 1] :: ld c, a` at
; lines 26-27 leaves caller's a == target's exit c.
;
; Why hl is clobbered pre-call: the `farcall` macro expands to
;   ld a, BANK(\1) :: ld hl, \1 :: rst FarCall
; If the target reads hl as input, this is a bug.
;
; Three valid call patterns:
;   1. push hl :: farcall TARGET :: pop hl
;        if hl just needs preserving across the call.
;   2. ROM0 `homecall` thunk
;        if target needs hl as input AND output.
;   3. Pass hl via bc/de and reconstruct inside the target.
;
; For a-return cross-bank functions in the target:
;   - mirror a into c at every ret in the target (one `ld c, a` per
;     ret), or
;   - return via HRAM, or
;   - use a HOME thunk that stashes the result.
;
; Audited by:
;   tools/audit/check_farcall_hl_clobber.py  ; release-smoke floor
;   tools/audit/check_farcall_a_clobber.py   ; release-smoke floor
;

; "RGBDS / asm gotchas" section.

What's suboptimal

The flow:

The original intent was: save hOAMUpdate, set it to 1 (or some specific value), do the delay logic, restore. With ld a, 1 commented out, the value is never changed. So:

  • Line 27 push af stores the same value that's already in [hl].
  • Line 31 ld [hl], a writes the value back to where it was read from.
  • Line 76 pop af restores the unchanged value.
  • Line 77 ldh [hOAMUpdate], a writes it back to the same address.

If the commented-out ld a, 1 line was intentionally disabled (i.e., the code is meant to NOT touch hOAMUpdate), then the save/restore plumbing is leftover scaffolding that can be deleted entirely — saves a few bytes and skips a few cycles per text-letter delay (which runs many times per battle/cutscene).
